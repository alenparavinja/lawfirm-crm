# Data Model

The database models a small immigration law firm. MongoDB is used with a
referenced (normalized) design rather than deep embedding, because the
application needs to query entities independently (for example, all open
cases across every client, or all documents of a given type) and those
queries are awkward against embedded arrays.

Collections are grouped into two tiers. Tier 1 is the core that the API and
frontend exercise directly and is seeded with realistic volumes. Tier 2 is
reference and scheduling data whose structure is defined here but which is
seeded lightly until the core application works end to end.

## Conventions

Every document has an `_id` generated by MongoDB. Cross-collection links are
stored as the referenced document's `_id`. Dates are stored as native BSON
dates. Enumerated fields (status, type, stage) use lowercase string values
listed below so the API and frontend can rely on a fixed vocabulary.

## Tier 1

### clients

A person the firm represents. Immigration clients carry fields a generic CRM
would not, which is what makes this domain specific.

Fields: full name, date of birth, country of origin, alien registration
number (A-number, format like A123456789), current immigration status (the
status or visa the client currently holds, for example "F-1 student" or
"none / undocumented"), date of entry into the country, email, phone,
mailing address, date onboarded as a client, and an active/inactive status.

### staff

Attorneys and paralegals at the firm. Cases, notes, and tasks reference staff.

Fields: full name, role (attorney, paralegal, admin), email, and for
attorneys a state bar number. A simple active flag tracks current employment.

### cases

An immigration matter for a client. This is the central entity. References
the client it belongs to and the staff member responsible for it.

Fields: a human-readable case number (for example "2024-0042"), title,
the client reference, the responsible staff reference, case type (the
immigration benefit being sought), the government receipt number assigned
by USCIS (format like MSC2190012345, may be empty before filing), priority
date, filing date, current stage, overall status, date opened, and date
closed when applicable. An optional court reference links the case to a
jurisdiction when it is in proceedings.

Case type values: employment_visa, student_visa, asylum, green_card,
naturalization, removal_defense, family_petition, other.

Stage values: consultation, preparing, filed, rfe_received, interview_scheduled,
approved, denied, appeal. These are the conventional vocabulary the application
uses; the field is stored as a free string and is not enforced by a schema enum.

Status values: open, closed, on_hold.

### documents

Metadata for a file attached to a case. Actual file contents are out of
scope; only metadata is stored. References the case it belongs to.

Fields: title, document type, the case reference, upload date, the author
(free text or a staff reference), a placeholder file path, and a file size
in bytes.

Document type values: passport, visa, application_form, supporting_evidence,
rfe_response, correspondence, court_filing, identity_document, other.

### notes

A timestamped note attached to a case. References the case and the staff
member who wrote it.

Fields: the case reference, the author staff reference, the note body as
free text, and a created timestamp.

### tasks

A to-do item, usually but not always linked to a case. References the
assigned staff member and optionally a case.

Fields: title, optional case reference, assigned staff reference, due date,
priority, and status.

Priority values: low, normal, high.

Status values: open, in_progress, done.

## Tier 2

These collections are defined now for completeness and seeded minimally.
They are mostly reference data that other entities link to.

### schedule

An appointment, hearing, or deadline. References a case and optionally a
court. Kept separate from tasks because immigration deadlines (filing
windows, hearing dates, biometrics appointments) are central to the practice
and benefit from their own queries and views.

Fields: the case reference, event type, title, start datetime, optional end
datetime, location, and an optional court reference.

Event type values: consultation, biometrics, interview, hearing,
filing_deadline, other.

### courts

Reference data for immigration courts and USCIS offices. Cases and schedule
entries link to these.

Fields: name, type, jurisdiction, and address.

Type values: uscis_field_office, uscis_service_center, immigration_court,
bia (Board of Immigration Appeals), circuit_court.

### statutes

Reference data linking to relevant immigration law. Cases may cite these.

Fields: citation (for example "INA 245" or "8 CFR 204.2"), title, a short
summary, and a URL to the authoritative source.

### case_law

Reference data for relevant precedent decisions.

Fields: case name, citation, deciding court, year, a short summary, and a
URL to the decision.

## Relationship summary

A client has many cases. A case has many documents, many notes, and many
schedule entries. A case belongs to one client and one responsible staff
member, and optionally links to one court. Tasks belong to one staff member
and optionally one case. Cases may cite statutes and case law from the
reference collections.
